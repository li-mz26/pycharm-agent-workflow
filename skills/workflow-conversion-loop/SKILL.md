# 工作流转换闭环（Workflow Conversion Loop）

## 何时使用
当用户提出以下需求时，必须使用本技能：
- 将现有工作流转换为另一种结构或风格；
- 通过 MCP 工具新建工作流；
- 修复执行失败或行为异常的工作流；
- 迭代提升工作流执行稳定性与质量。

只要涉及节点、连线或配置改动，本闭环就是**强制流程**。

## 不可跳过的执行顺序
每次“有意义的改动”后，必须按以下顺序执行：

1. **创建/修改** 工作流（JSON、nodes、edges、代码/提示词/配置文件）。
2. 立刻执行 **`workflow_validate`**。
3. 立刻执行 **`workflow_run`**（使用最小但真实的输入样例）。
4. **检查日志**（`logs`、`validationErrors`、工具报错、堆栈信息）。
5. 若失败或可疑，先做**单节点调试**（隔离单节点/单路径），修复后回到第 2 步。
6. 重复以上流程，直到同一输入下连续两次运行稳定。

禁止“只生成/只改文件，不做校验和运行”。

## 每轮必须回答的问题
每一轮迭代都必须明确回答：
- 校验是否通过？若未通过，具体是哪个字段/节点/边失败？
- 运行是否通过？若未通过，首个失败节点是谁？
- 错误日志原文核心信息是什么？
- 问题属于：数据映射、节点配置、路由连线、还是运行时依赖？

## 单节点调试协议
当运行失败时：
1. 从日志定位首个失败节点；
2. 核对该节点必需输入是否存在、字段映射是否正确；
3. 核对文件引用路径是否存在（`codeFile`、`promptFile`、配置路径）；
4. 临时简化分支/条件路由，强制仅走失败路径；
5. 复跑确认节点恢复后，再还原完整图。

优先做最小改动，快速复跑验证。

## 分支与路由防错约束
- 路由标签与逻辑保持一致（例如统一使用 `true/false` 或 `有数据/无数据`，不要混杂歧义标签）。
- `branchField` 必须能在运行时输入/输出结构中被正确解析。
- 任何路由改动后，至少覆盖每条关键分支一次运行。

## 代理输出格式要求
每轮迭代结束后，必须汇报：
- 改动了什么（节点/连线/配置/文件）；
- `workflow_validate` 结果摘要；
- `workflow_run` 结果摘要；
- 关键错误日志或成功确认日志；
- 下一步动作（继续闭环或结束）。

## 完成标准（Done Criteria）
仅当以下条件全部满足时，工作流转换任务才可结束：
- `workflow_validate` 通过；
- `workflow_run` 成功；
- 关键分支已被覆盖运行（若未覆盖必须明确说明）；
- 无未处理的错误日志。

任一条件不满足，必须继续闭环迭代。
